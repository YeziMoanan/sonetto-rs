# Sonetto RS 分阶段开发计划

> 状态：需求补充草案  
> 更新日期：2026-07-14  
> 目标仓库：`sonetto-rs`  
> 配套仓库：`sonetto-patch`（仅在协议拦截或客户端兼容确有需要时修改）

## 1. 文档用途

本文件是项目级路线图，用于收集需求、固定开发顺序并跟踪阶段状态。每个阶段开始前，另行编写一份详细实施计划，精确列出修改文件、测试用例、执行命令和提交边界。

需求可直接补充到对应阶段的“需求补充清单”，建议采用以下格式：

```text
- [REQ-阶段编号-序号] 场景：玩家执行什么操作
  - 预期：客户端和服务端应出现什么结果
  - 优先级：必须 / 应该 / 可选
  - 备注：协议样本、截图、日志或特殊规则
```

## 2. 已确认原则

- 按 README 的未完成功能顺序开发。
- 一个阶段只处理一个独立子系统；阶段内拆成可单独验证的任务。
- 每个行为变更先写失败测试，再写最小实现。
- 每个任务完成后运行局部测试；每个阶段完成后运行工作区完整检查。
- SQLite 仍是唯一运行时数据库，不引入外部数据库服务。
- 保持现有客户端协议兼容，不随意修改生成的 `protocol/include/_.rs`。
- 普通副本、登录、背包、商店、抽卡和邮件等已工作流程不得回退。
- 已在 README 中划除的“英雄天赋持久化”和“完整货币逻辑”先做回归验证，不重新列为开发阶段。
- 不修改系统代理、环境代理或任何代理配置。

## 3. 交付与状态规则

阶段状态使用以下值：

- `待补充需求`：范围尚未确认。
- `待设计`：需求已确认，尚未形成详细方案。
- `待开发`：详细计划已确认。
- `开发中`：正在按任务实施。
- `待验收`：自动化检查通过，等待实际客户端验证。
- `完成`：验收标准全部满足。

默认交付方式：每个阶段使用独立功能分支，每个任务形成一个聚焦提交。未经明确要求，不推送分支或创建拉取请求。

## 6. 阶段 1：爬塔战斗

### 已确认现状

- `StartTowerBattleCmd` 已注册并能创建战斗。
- `ActiveBattle` 已保存 `tower_type`、`tower_id` 和 `layer_id`。
- SQLite 已包含塔、层、关卡、阵容和辅助 Boss 数据表。
- starter data 当前包含 10 个开放记录、26 个塔记录、3 种塔类型和 6 个辅助 Boss。
- 通用 `BeginRound` 结算忽略塔标识，错误地更新普通副本进度。
- `TowerBattleFinishPush` 与 `TowerLayerUpdatePush` 已存在于协议中，但当前没有发送。
- 数据库中已有塔层分数、关卡状态和通关层更新函数，但当前结算流程没有调用。

### 第一批目标：基础爬塔闭环

玩家可以从现有塔数据进入关卡，完成战斗后获得塔专用结算；关卡状态、层分数和通关进度写入 SQLite，退出并重新登录后仍然保留。

### 任务

- [ ] 1.1 为塔进度查询和结算建立数据库集成测试。
- [ ] 1.2 校验开战请求中的塔类型、塔 ID、层 ID 和关卡 ID 属于当前用户可用数据。
- [ ] 1.3 明确区分普通副本战斗与爬塔战斗的结算路径。
- [ ] 1.4 使用单个数据库事务更新关卡状态、当前分数、历史最高分和已通过层。
- [ ] 1.5 发送 `TowerBattleFinishPush` 和 `TowerLayerUpdatePush`，不再发送错误的普通副本进度通知。
- [ ] 1.6 战斗失败或主动退出时不写入通关进度。
- [ ] 1.7 重新查询塔信息，验证持久化结果与结算通知一致。
- [ ] 1.8 添加普通副本结算回归测试，防止共享流程被破坏。
- [ ] 1.9 使用实际客户端完成进入、战斗、结算、重登四步验收。
- [ ] 1.10 更新 README 的功能状态和已知限制。

### 第一批验收标准

- 有效塔关卡能够进入战斗。
- 胜利后客户端收到塔专用结算和层更新通知。
- 胜利只更新对应用户、塔、层和关卡，不修改普通副本记录。
- 当前分数、历史最高分和通关层按已确认规则更新。
- 失败、主动退出、重复结算和非法关卡请求不会错误增加进度。
- 重启服务并重新登录后，`GetTowerInfoCmd` 返回更新后的进度。
- 普通副本原有开战与结算测试继续通过。

### 暂不纳入第一批

- 扫荡。
- 辅助 Boss 天赋激活、重置、改名和方案切换。
- 子关卡重置。
- 深层塔存档、读取、重置与结算。
- 定时赛季轮换。

### 需求补充清单

请优先补充以下规则，它们会直接决定详细实施方案：

- 分数如何计算；是否可先使用固定分数完成协议闭环。
- `status` 各数值代表未解锁、可挑战、已完成中的哪一种状态。
- 一层包含多个关卡时，满足什么条件才更新 `pass_layer_id`。
- 胜利后是否发放奖励；奖励从哪份配置读取。
- 历史最高分是否只增不减，当前分数是否允许被较低分覆盖。
- 塔类型 `1`、`2`、`3` 是否在第一批采用相同结算规则。
- 失败、主动退出、断线和重复结算分别应如何处理。
- 客户端版本与可复现入口，包括塔 ID、层 ID、关卡 ID 和操作步骤。

## 14. 每阶段完成定义

一个阶段只有同时满足以下条件才能标记为“完成”：

- 需求与明确排除项已经记录。
- 新行为具有先失败、后通过的自动化测试。
- 局部测试、`cargo fmt --check`、`cargo check --workspace` 和 `cargo test --workspace` 全部通过。
- SQLite 迁移可用于新数据库，并验证旧数据库升级路径。
- 错误请求不会造成部分写入、重复奖励或跨用户数据污染。
- 使用实际客户端完成主要成功路径和至少一个失败路径的验收。
- README 中的功能状态与已知限制已经更新。
- 改动拆成可审查的聚焦提交，且工作区不包含无关文件。
