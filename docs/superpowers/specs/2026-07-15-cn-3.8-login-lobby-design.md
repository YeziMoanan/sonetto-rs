# 国服 3.8 登录与大厅适配设计

## 1. 目标

以当前可运行的国际服 3.6 `sonetto-rs` 工作树为基底，建立独立的国服 3.8 服务端副本，并使指定的国服 3.8 Android 客户端能够重复完成访客登录、选服、建立游戏连接并进入大厅。

本设计的首个里程碑只覆盖登录和大厅必需链路。角色养成、完整背包操作、抽卡、活动和战斗等功能在进入大厅后按独立阶段继续适配。

## 2. 固定输入

- 目标 APK：`D:\python-tools\重返未来1999\重返未来：1999.apk`
- 包名：`com.shenlan.m.reverse1999`
- 版本名：`3.8.0`
- 版本码：`190`
- APK SHA-256：`EA6CD8AD7FAAFE6EDA42F4C2073DCF1BDA7F24AAE9A7011FDFE918AFFF69D3C0`
- APK 签名证书 SHA-256：`07F44EA068873C1220D9471839D0F6A830636C4BBE3690EFD665A27AB848C20F`
- APK 内部渠道：`channelId=100`、`subChannelId=1000`、`sdkEnv=shenlan`
- MuMu 专用实例：索引 `2`，名称 `R1999-3.8-CN-Dev`
- 3.6 服务端源目录：`D:\python-tools\重返未来1999\sonetto-rs`
- 3.8 服务端目标目录：`D:\python-tools\重返未来1999\sonetto-rs-3.8-cn`
- 3.8 HTTP 端口：`21100`
- 3.8 游戏 TCP 端口：`23401`

MuMu ADB serial 是运行时临时值，不写入源码、配置或规格。所有设备诊断都必须从实例索引 `2` 刷新当前 endpoint。根据项目级安全约束，后续工作只使用非交互文件与终端操作，不执行系统控制或界面控制。

## 3. 安全与操作约束

- 不修改 Windows、WinHTTP、环境变量或 Android 的代理设置。
- 不修改 DNS、hosts、证书或全局网络配置。
- 不运行全局 `adb kill-server`，不干扰其他 ADB 会话或 MuMu 实例。
- 客户端流量只允许按精确 allowlist 重定向到本机 loopback 端口。
- 不绕过支付、反作弊、账号鉴权或访问控制。
- 日志默认不记录 Authorization、Cookie、账号令牌、设备令牌和完整请求体。
- 不复制或修改国际服 3.6 的运行数据库。
- 不向远端推送分支或创建拉取请求，除非用户另行明确要求。

## 4. 不在首个里程碑内

- 同时支持 `Reverse1999_Bluepoch_1000.apk` 的版本码 `130` 构建。
- 完整实现所有 3.8 命令和业务系统。
- 完整复刻官方资源更新、支付、公告和风控服务。
- 修改或重新签名官方 APK。
- 建立长期多版本共用的协议抽象层。
- 自动执行 MuMu 界面操作或代替用户完成客户端验收。

## 5. 基线隔离

### 5.1 副本建立

`sonetto-rs-3.8-cn` 使用本地 Git 克隆保留 `sonetto-rs` 的提交历史，并在独立分支 `feat/cn-3.8-login-lobby` 上开发。克隆完成后，将当前 3.6 工作树中的有效源码和必要的未跟踪开发文件覆盖到副本，以保留尚未提交但已经参与可运行 3.6 基线的改动。

覆盖前后生成文件清单和 SHA-256 清单，明确哪些文件来自已提交历史、当前工作树或后续 3.8 开发。

### 5.2 复制范围

复制以下内容：

- Cargo 工作区源码、清单和锁文件。
- 当前协议容错、包处理和连接状态改动。
- 当前 SDK 访客登录与路由实现。
- 配置生成器及其实际使用的生成配置源码。
- 与构建、测试和数据生成直接相关的脚本、工具和文档。

排除以下内容：

- 源目录的 `.git`，目标目录保留本地克隆产生的 `.git`。
- `target`。
- `runtime/db`、根目录 `sonetto.db` 和所有 SQLite 备份。
- `*.log`、`*.err.log`、`*.bak` 和临时构建输出。
- 任何进程 ID、临时 ADB serial 或本机运行会话数据。

### 5.3 数据隔离

3.8 使用新的运行目录和全新 SQLite 数据库。第一阶段可以复用 3.6 静态表完成服务启动和协议探测，但不得把它们标记为 3.8 最终数据。

客户端资源下载完成后，从指定版本码 `190` 构建及其已下载资源建立独立的 `3.8-cn` 数据集。3.8 配置生成结果不得覆盖 `sonetto-data` 中现有的国际服 3.6 数据。

## 6. 总体架构

```text
国服 3.8 APK（MuMu 实例 2）
  |
  | 精确端点重定向，不修改代理
  v
ADB loopback 映射 / 现有 Android Hook 工程
  |                         |
  | HTTP :21100             | TCP :23401
  v                         v
3.8 sdkserver            3.8 gameserver
  |                         |
  | 版本、资源、登录、选服   | 握手、包头、命令与大厅初始化
  +------------+------------+
               |
               v
       独立 SQLite 与 3.8 数据集
```

Android 端接入继续放在现有 `sonetto-patch` 仓库中，并绑定目标 APK 哈希。服务端适配只放在 `sonetto-rs-3.8-cn` 中。两者通过配置化端口和脱敏日志交换证据，不共享硬编码进程地址或临时设备信息。

## 7. 登录与协议数据流

### 7.1 SDK 阶段

按客户端实际请求顺序适配以下能力：

1. 版本检查和资源清单。
2. SDK 初始化与登录配置。
3. Shenlan 访客登录。
4. 区服列表与区服选择。
5. `start_game` 和游戏会话票据。

每个接口先记录脱敏后的请求形状、现有 3.6 响应和 3.8 客户端行为，再决定最小响应。未实现接口返回结构化错误，不伪造成功。

### 7.2 游戏连接阶段

客户端连接 `gameserver` 后依次确认：

1. TCP 连接目标和握手顺序。
2. 包头长度、端序、命令 ID、序列号和状态码位置。
3. 压缩标志与压缩算法。
4. 拆包、粘包和多响应顺序。
5. 登录命令与大厅初始化命令序列。

已知命令进入现有处理器或 3.8 专用适配器。未知命令只记录命令 ID、负载长度、连接阶段和解析结果，不允许单个未知包直接使服务端进程退出。

### 7.3 大厅初始化

大厅初始化只返回客户端进入主界面所需的最小一致状态，包括玩家基础信息、角色摘要、货币与背包摘要、功能开关、红点或活动占位信息，以及客户端明确等待的其他初始化响应。

所有初始化写入使用独立数据库事务。任一必需步骤失败时完整回滚，避免产生半初始化账号。重复访客登录和服务重启必须返回一致状态。

## 8. 协议与静态数据策略

- 不凭猜测直接编辑生成的 `protocol/include/_.rs`。
- 优先从 3.8 客户端元数据、真实命令日志和可复现请求样本确定字段。
- 每个新增命令保留脱敏后的输入样本、预期输出和最小回归测试。
- 3.8 命令 ID、结构定义和配置生成结果与 3.6 分开保存。
- 3.6 数据只用于引导服务启动；凡是影响大厅显示或客户端校验的数据，都必须用 3.8 数据替换后才算完成。
- 如果 3.8 与 3.6 包格式兼容，复用现有解析器；如果不兼容，只在明确边界增加版本专用解析器，不提前重构整个多版本协议层。

## 9. 故障处理与可观测性

- 启动或采集前校验 APK 版本、哈希和签名；不匹配时停止该流程。
- Hook 只重定向已确认 allowlist 中的游戏端点，其他流量保持原样。
- SDK 错误日志包含路径、HTTP 状态、请求形状和关联 ID，不包含凭据。
- TCP 错误日志包含时间、连接阶段、命令 ID、负载长度和解析错误。
- 单连接协议错误只关闭该连接；共享服务状态不得因一个坏包损坏。
- 数据库迁移、账号初始化和大厅状态更新使用事务并验证回滚。
- 3.8 日志、数据库、端口和运行目录与 3.6 完全分离。
- 3.8 服务启动失败时，不修改或回退 3.6 目录中的任何文件。

## 10. 测试策略

### 10.1 基线测试

- 比较 3.6 源目录与 3.8 副本的复制清单。
- 验证所有排除项没有进入新副本。
- 记录带入副本的未提交源码改动和必要未跟踪文件。

### 10.2 构建与单元测试

- `cargo fmt --check`
- `cargo check --workspace`
- `cargo test --workspace`
- SDK 路由、响应结构和错误路径测试。
- 包头、端序、压缩、拆包、粘包和未知命令测试。
- 新账号初始化、事务回滚、重复登录和重启恢复测试。

### 10.3 本地集成测试

在 `127.0.0.1:21100` 和 `127.0.0.1:23401` 启动 3.8 服务，使用 HTTP 与 TCP 探针验证：

- 两个端口只监听 loopback。
- SDK 请求能够完成规定的登录序列。
- 游戏连接能够完成握手并进入大厅初始化阶段。
- 未知或畸形请求不会使服务端进程退出。

### 10.4 客户端验收

客户端验收由用户在 MuMu 实例 `2` 中手动执行。服务端侧同时保存脱敏日志，验证客户端确实完成：

1. 启动指定版本客户端。
2. 完成访客登录。
3. 选择区服。
4. 建立游戏 TCP 连接。
5. 进入大厅并保持稳定。
6. 关闭并重新启动客户端后重复上述流程。

## 11. 实施阶段

### 阶段 0：建立独立基线

创建 `sonetto-rs-3.8-cn`、复制当前有效工作树、生成清单、建立独立配置和空数据库，并完成工作区构建检查。

### 阶段 1：SDK 登录链路

适配版本、资源、SDK 初始化、访客登录、区服和 `start_game`，通过 HTTP 契约与集成测试。

### 阶段 2：TCP 与协议识别

建立精确重定向和本地链路，确认 3.8 包格式、命令 ID、压缩和登录命令顺序。

### 阶段 3：大厅初始化

补齐进入大厅需要的最小响应、独立账号状态和重登一致性。

### 阶段 4：实际客户端验收

由用户手动完成首次登录和重登，服务端日志证明请求闭环，并确认国际服 3.6 目录和行为未改变。

## 12. 完成定义

首个里程碑只有同时满足以下条件才算完成：

- 目标 APK 的版本、哈希和签名均匹配固定输入。
- 3.8 服务可在独立端口和独立数据库上重复启动。
- SDK 登录、选服和会话票据链路完整。
- 游戏 TCP 连接完成握手并完成大厅必需初始化。
- 用户确认客户端成功进入大厅，重启后可重复。
- 未知请求和单连接错误不会导致服务端进程崩溃。
- 3.8 工作区构建、单元测试和集成测试通过。
- 国际服 3.6 源目录、数据库、端口配置和运行行为未被修改。
